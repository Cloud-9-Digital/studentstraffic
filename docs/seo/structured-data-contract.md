# Structured-data contract for destination, university and program pages

Updated: 2026-07-11

This document defines the JSON-LD contract for Students Traffic's three core catalog page
families:

- country pages: `/countries/{country}`;
- university pages: `/university/{university}`;
- individual program pages: `/{program-offering}` and its section variants such as
  `/{program-offering}-fees`.

The implementation is centralized in [`lib/structured-data.ts`](../../lib/structured-data.ts) and
rendered through [`components/shared/json-ld.tsx`](../../components/shared/json-ld.tsx). Page routes
compose a single `@graph`; the root layout supplies site-wide entities.

## Principles

1. Mark up what is visible and verifiable on the page. JSON-LD must not introduce claims that are
   absent from the public page or unsupported by internal provenance.
2. Prefer stable entity references. Every country, course, university and program offering has a
   deterministic `@id` fragment so related graphs refer to the same entity.
3. Use the most specific entity available. A university is `CollegeOrUniversity`; an individual
   offering is `Course`; a country is `Country`; a listing page is `CollectionPage`.
4. Do not add fake reviews, ratings, authors, rankings, accreditation, fees or outcomes. Missing
   facts are omitted rather than guessed.
5. Keep the graph useful even when optional content is empty. FAQ and program-list nodes are only
   emitted when their source data exists.
6. Sanitize JSON-LD before rendering. The shared `JsonLd` component escapes `<` to prevent markup
   injection, following the Next.js JSON-LD guidance.

## Site-wide graph

The root layout emits these entities on every page:

### `Organization`

Source: `siteConfig` in `lib/constants.ts`.

Fields include:

- `@id`: `https://studentstraffic.com/#organization`;
- `name`, `alternateName`, `url`, `logo`, `image`;
- support `email`, `telephone`, `contactPoint` and `areaServed`;
- site description.

### `WebSite`

Fields include:

- `@id`: `https://studentstraffic.com/#website`;
- site name, URL, description and language;
- `publisher` pointing to the site `Organization`;
- `SearchAction` targeting the site search URL.

### `Person`

The governed content author is emitted with the site-wide graph. Author identity must come from
the maintained author configuration; do not create a person solely to fill a schema field.

## Country-page graph

Route: `/countries/{slug}`.

The page emits:

1. `BreadcrumbList`
2. `Country`
3. `CollectionPage`
4. `ItemList` of currently published program offerings, when available
5. `FAQPage`, when country FAQs are available
6. `Article` metadata using the governed author and page review date

### `Country`

Generated by `getCountryStructuredData(country)`.

Required source fields:

- `country.slug` for the canonical URL and `@id`;
- `country.name`;
- `country.summary`;
- the country hero/OG image resolver.

Output shape:

```json
{
  "@type": "Country",
  "@id": "https://studentstraffic.com/countries/vietnam#country",
  "name": "Vietnam",
  "description": "...",
  "url": "https://studentstraffic.com/countries/vietnam",
  "image": "https://studentstraffic.com/...",
  "mainEntityOfPage": {
    "@id": "https://studentstraffic.com/countries/vietnam#webpage"
  }
}
```

### `CollectionPage`

The country page itself is a destination-level collection page. Its description must remain
country-level even when the current catalog is dominated by one course. Its `mainEntity` points to
the country `ItemList` when published programs exist.

### Country `ItemList`

Each list item is a `ListItem` whose `item` is a program-offering `Course`. The list must use only
published offerings returned by the catalog query. It must not include draft, held or unpublished
programs.

## University-page graph

Route: `/university/{slug}` and supported university section variants.

The page emits:

1. `BreadcrumbList`
2. related `Country`
3. related catalog `Course` for the primary offering, when available
4. `WebPage`
5. `CollegeOrUniversity`
6. `ItemList` of the university's published program offerings, when available
7. `FAQPage`, when university FAQs are available
8. separately rendered genuine `Review` and `AggregateRating` nodes when live reviews qualify
9. `Article` metadata using the governed author and page review date

### `CollegeOrUniversity`

Generated by `getUniversityStructuredData({ university, country, programs })`.

Fields are derived from the university catalog record:

- `@id`, `name`, `description`, `url`;
- indexed `image` URLs and `logo` when present;
- `sameAs` from the official website and verified recognition links;
- `foundingDate` from `establishedYear`;
- `PostalAddress` with city and country;
- `mainEntityOfPage` pointing to the university `WebPage`;
- `hasCourse` for published program offerings.

The institution's `sameAs` array must contain real canonical profiles or official verification
URLs. Recognition copy belongs in visible page content and internal provenance; a recognition URL
alone must not be treated as proof of an accreditation claim.

### Reviews and ratings

The review component emits `Review` nodes only for live text reviews with a body. A review may have
a `Rating` only when the submitted review has a star rating. An `AggregateRating` is emitted only
when at least one qualifying rated review exists, and it is calculated from those reviews.

The review implementation must continue to filter out hidden, archived and otherwise non-live
records through `getUniversityReviews`. Never import a third-party rating such as Collegedunia's
rating into Students Traffic's university schema.

## Individual program-page graph

Route: `/{program-offering}` and section variants.

The individual program route is different from `/courses/{course}`. It describes one university's
specific offering and therefore emits:

1. `BreadcrumbList`
2. related `Country`
3. catalog `Course` for the course family
4. `CollegeOrUniversity` for the provider institution
5. `WebPage` for the specific program URL
6. program-offering `Course`
7. `CourseInstance` and a paid `Offer` when the published fee exists
8. `Article` metadata using the governed author and page review date
9. `FAQPage` using the visible university FAQs when available

### Program-offering `Course`

Generated by `getProgramOfferingStructuredData(program)`.

Output includes:

- stable `@id` based on `program.offering.slug`;
- official offering title and canonical program URL;
- description identifying course, university, city and country;
- `provider` pointing to the university entity;
- course code and credential name from the canonical course;
- `courseMode` as `Onsite` for the current catalog's campus-based offerings;
- `inLanguage` as currently supported by the catalog contract;
- `timeRequired` from the offering duration;
- `Offer` with normalized USD annual tuition only when a published fee exists;
- `mainEntityOfPage` pointing to the program `WebPage`.

The normalized USD amount is an analytical display value. It must not replace the official native
currency amount in visible page content or provenance. Do not emit an `Offer` when the fee is
missing, stale or unpublished.

### `Article`

University, country, course-family and individual program pages are editorial catalog guides with
real maintained content dates, so they emit `Article` metadata. The article node references the
site's governed author `Person`, the site `Organization` publisher, the page as
`mainEntityOfPage`, and the page's current image. It does not claim a separate author or date that
is not present in the content-governance configuration.

### Program page versus course page

`/courses/{slug}` remains a course-family guide. It emits the canonical catalog `Course`, a
`CollectionPage`, an `Article`, and an `ItemList` of representative offerings. It must not be
confused with the individual offering route, which uses the offering slug and university provider.

## Breadcrumb rules

Breadcrumbs are generated from the real route hierarchy:

- country: Home → Guides → Countries → Country;
- university: Home → Universities → University;
- individual program: Home → Universities → University → Course/program.

The final breadcrumb URL must equal the page's canonical path. Do not add duplicate Home entries,
section labels that are not navigable, or URLs that redirect unnecessarily.

## FAQ rules

`FAQPage` is emitted only when the questions and answers are actually rendered or otherwise clearly
available on the page. Answers must be complete, factual and sourced internally. The content
standard targets 10–13 university FAQs; country and program FAQ counts may be lower when the topic
does not apply, but empty or placeholder answers must never be emitted.

## Data requirements by page type

| Page | Minimum data | Optional data |
|---|---|---|
| Country | slug, name, summary, URL, image, published programs | FAQs, richer country media |
| University | identity, country, city, type, year, official URL, summary | logo, images, recognition links, programs, FAQs, reviews |
| Program | offering slug/title, university, country, course, duration, medium, official URL | verified fee, native fee, intake, FAQs |

## Validation checklist

For every page family:

1. Confirm exactly one JSON-LD graph is emitted by the route in addition to the site-wide graph.
2. Confirm every `@id` is absolute, stable and unique for its entity.
3. Confirm all internal references resolve to an entity in the graph or the site-wide graph.
4. Confirm canonical page URL, breadcrumb final URL and `WebPage.url` agree.
5. Confirm all listed programs are published and attached to the right university/country/course.
6. Confirm FAQ questions match visible page content.
7. Confirm ratings and reviews are first-party, live and mathematically consistent.
8. Confirm missing optional facts are omitted rather than represented by empty strings or guesses.
9. Run the TypeScript checks and the automated structured-data tests.
10. Spot-check the rendered HTML with the Schema Markup Validator and Google Rich Results Test.

Google's Rich Results Test is useful for Google-supported enhancements; the Schema Markup Validator
is the broader check for Schema.org validity. Passing either tool does not guarantee rankings or a
rich result.

## Change procedure

When a new schema entity or property is added:

1. Update `lib/structured-data.ts` and keep the shape shared across routes.
2. Add or update tests for the entity's `@type`, `@id`, required relationships and omission rules.
3. Update this document with source fields, output fields and page ownership.
4. If the data model or publishing payload changes, update the university pipeline and content
   specification in the same change.
5. Run typecheck, tests and a rendered-page inspection before release.
